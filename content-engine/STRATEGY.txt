BEYOND OPEN RATE — CONTENT STRATEGY
LinkedIn + Blog
=====================================

OVERVIEW
--------
Two automated content pipelines, both with a human review gate before
anything publishes:

1. LinkedIn posts — frequent, short-form, top-of-funnel awareness
2. Blog posts — less frequent, long-form, SEO-driven

Both are generated by Claude via GitHub Actions, reviewed by Tim in a
PR, and published automatically on merge.


=====================================
1. LINKEDIN STRATEGY
=====================================

CADENCE
Monday, Wednesday, Thursday, Friday at 8am AEST.
4 posts/week — a GitHub Action picks the next topic from the queue,
generates two copy variants, and opens a PR for review.

POSITIONING (what every post reinforces)
Beyond Open Rate sits between cheap unreliable freelancers and bloated
enterprise agencies. Every deliverable ties to revenue — not opens,
clicks, or engagement. Platform-agnostic: no platform names are ever
mentioned in posts, and recommendations are based on what's right for
the business, not commission.

AUDIENCE
E-commerce founders / DTC operators doing $1M–$20M/year revenue who are:
- Time-poor
- Possibly burned by a generic agency or freelancer before
- Quietly aware email is leaking money but haven't acted
- Not email nerds — they care about revenue, not tactics
The audience is COLD. They don't know Beyond Open Rate and aren't
actively looking for email help. Most don't think they have a problem yet.

COPYWRITING FRAMEWORK (Sabri Suby "Sell Like Crazy" principles)
1. Lead with pain or cost, never the solution.
   Wrong: "Email lifecycle is underutilised..."
   Right: "You paid $30 to acquire that customer. You sent one welcome
   email. Then silence."
2. Match the market-temperature tier (see below).
3. Emotional triggers over logic — primarily MONEY already earned and
   nearly lost, secondarily TIME they know they should spend but can't.
4. Specificity builds credibility.
   Wrong: "Most brands leave money on the table."
   Right: "Brands doing $2–5M recover under 8% of abandoned carts.
   Top performers recover 22%."
5. Short sentences, one idea per line — LinkedIn rewards scannable copy.

MARKET-TEMPERATURE TIERS (and the target mix per 8 posts / fortnight)
- unaware        (60% of market) → 2 posts — pattern interrupt, no CTA
- problem_aware  (20%)            → 3 posts — education, soft CTA
- gathering_info (17%)            → 2 posts — authority/methodology, audit CTA
- ready_to_buy   (3%)              → 1 post  — direct offer, used sparingly

CTA STYLES
- none   — pure value, no ask (unaware tier)
- soft   — "follow for more" / engagement prompt (problem_aware)
- audit  — link to the free email program audit tool (gathering_info)
- direct — book a call (ready_to_buy, max 1 in 8–10 posts)

TONE RULES
- Direct, a bit blunt. Sounds like a practitioner, not a content creator.
- Confident without arrogance. Australian spelling, no Americanisms.
- Banned: "Unpopular opinion:", "Hot take:", emoji bullets, "game-changer",
  "holistic", "synergy", "leverage" (verb), "unlock", rhetorical-question
  openers, 10+ item lists, generic "What do you think?" engagement bait,
  anything that reads as AI-generated, generic advice any agency could write.
- Format: no headers/bold, line breaks between thoughts, 200 words typical
  (300 hard cap), CTA as a single line at the end after a blank line.

WORKFLOW
1. GitHub Action (Mon/Wed/Thu/Fri 8am AEST) pulls next "queued" topic
   from content-engine/topics.json, generates 2 variants via Claude,
   opens a PR with the draft MDX in content-engine/posts/drafts/.
2. Tim reviews both variants in the PR, edits if needed, sets frontmatter:
   approved_variant ("A" / "B" / "EDITED"), schedule_time (ISO 8601 +10:00),
   optionally facebook: true + facebook_variant for repurposing.
3. On PR merge → publish workflow fires → content-engine/scripts/publish-post.js
   schedules the approved copy to LinkedIn via Buffer's GraphQL API,
   moves the file to posts/published/, logs it in posted.json.

TOPIC QUEUE MANAGEMENT
Maintained in content-engine/topics.json. Each topic has: id, status,
tier, topic, seed_claim (the actual POV/argument — not generic),
audience_angle (who exactly this is for and what they currently believe),
cta_style, notes. Aim to keep 4–6 weeks of topics queued at all times —
one ~20 minute brainstorm every 3–4 weeks (15–20 topics) is enough to
stay ahead.

STACK
Claude (generation) → GitHub Actions (orchestration + PR review gate)
→ Buffer GraphQL API (scheduling) → LinkedIn (+ optional Facebook).


=====================================
2. BLOG STRATEGY
=====================================

CADENCE
Tuesdays and Fridays at 9am AEST — 2 posts/week, SEO-driven long-form.

GOAL
Long-form, search-intent content that ranks for specific email-marketing
keywords e-commerce founders/operators actually search (e.g. "klaviyo
flows ecommerce", "email list cleaning", "spf dkim dmarc explained").
Unlike LinkedIn (cold pattern-interrupt), blog content targets people
already actively searching — closer to gathering_info / problem_aware
intent.

VOICE
- Senior email specialist explaining something to a smart client
- Short paragraphs (2–4 sentences), no filler intros
  ("In today's digital landscape...")
- Open directly with the problem or the stakes
- Concrete numbers and examples wherever possible
- Australian spelling ("optimise", "colour"; "email program" as a term
  of art is fine)

STRUCTURE (enforced via prompt template)
- Frontmatter: title, description (140–160 chars incl. primary keyword),
  publishedAt, author, tags (2–4), and a 3-item FAQ block (real
  Google-style questions + 2–4 sentence answers) — this also feeds an
  FAQPage schema for SEO.
- Opening paragraph states the problem/stakes immediately, no preamble.
- 4–7 ## sections, ### subsections where genuinely warranted.
- **Bold** key terms on first use. Lists only where genuinely list-like.
- 800–1,200 words of body content.
- At least 2 internal links worked in naturally:
  /tools/email-audit (free audit tool), /#contact (once, near the end),
  /blog (when referencing other posts generically).

WORKFLOW
1. GitHub Action (Tue/Fri 9am AEST) pulls the next "queued" entry from
   content/blog-queue.json (id, title, keyword, tags, notes), calls
   Claude to generate the full MDX file, opens a PR on a new
   blog/<slug> branch.
2. Tim reviews against a checklist: tone/voice, factual accuracy,
   internal links present, FAQ questions are real search queries,
   description under 160 chars.
3. Merge → Vercel deploys automatically. No separate publish workflow
   needed — it's just markdown content in the Next.js site.

TOPIC QUEUE MANAGEMENT
content/blog-queue.json — each entry has id (slug), title, keyword
(primary SEO target), tags, notes (angle/structure guidance for Claude),
status (queued / published).

STACK
Claude (generation) → GitHub Actions (orchestration + PR review gate)
→ Vercel (auto-deploy on merge to main).


=====================================
3. WHY TWO SEPARATE ENGINES
=====================================
LinkedIn and blog serve different funnel stages and different
distribution mechanics, so they're deliberately NOT unified:

- LinkedIn = push, cold audience, pattern-interrupt and authority-building,
  short-form, tier-based persuasion mix, scheduled via Buffer.
- Blog = pull, warm/search-intent audience, comprehensive answers to
  specific questions, long-form, SEO-structured, deployed via Vercel.

Both share the same human-review-gate pattern (Claude drafts → GitHub PR
→ Tim approves → auto-publish on merge) so nothing goes out without a
human checking it first, but the prompts, cadence, voice rules, and
distribution channels are tailored to each format.
